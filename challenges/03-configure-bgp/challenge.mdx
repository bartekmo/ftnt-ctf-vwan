For the FortiGate NVAs in the vWAN hub to route traffic, they must establish BGP sessions with the Azure vWAN Virtual Router. This challenge requires you to configure BGP on both NVAs so that both sessions show as **Connected** in Azure.

## Overview

The Azure vWAN hub runs a Virtual Router with two BGP peer endpoints (one per availability zone). Each FortiGate NVA must establish a BGP session with both endpoints to provide redundancy.

```
FortiGate NVA 1 ──[eBGP]──► Hub BGP Peer 1 (AS 65515)
                └─[eBGP]──► Hub BGP Peer 2 (AS 65515)

FortiGate NVA 2 ──[eBGP]──► Hub BGP Peer 1 (AS 65515)
                └─[eBGP]──► Hub BGP Peer 2 (AS 65515)
```

## Steps

### 1. Find the hub BGP peer addresses

In the Azure Portal, navigate to:

**Virtual WANs → your vWAN → Hubs → your hub → BGP Peers**

Note the two peer IP addresses (typically in the `192.168.x.x` range) and their AS number (65515).

### 2. Configure BGP on NVA 1

Open the management interface of NVA 1 at <EnvVarInline field="url_fgt_nva1" />

Navigate to **Network → BGP** and configure:

| Parameter | Value |
|-----------|-------|
| Router ID | NVA management IP |
| Local AS | <EnvVarInline field="fgt_asn" /> |

Add two BGP neighbors:

| Remote IP | Remote AS | Description |
|-----------|-----------|-------------|
| Hub BGP Peer 1 IP | 65515 | vWAN-hub-peer1 |
| Hub BGP Peer 2 IP | 65515 | vWAN-hub-peer2 |

Enable **eBGP Multipath** to use both paths simultaneously.

### 3. Configure BGP on NVA 2

Repeat the identical configuration on NVA 2 at <EnvVarInline field="url_fgt_nva2" />

### 4. Configure static routes

Before BGP can establish sessions, the NVAs need static routes to reach the hub BGP peer addresses. In the FortiGate CLI:

```
config router static
    edit 1
        set dst <hub-bgp-peer-1>/32
        set gateway <nva-default-gateway>
        set device port2
    next
    edit 2
        set dst <hub-bgp-peer-2>/32
        set gateway <nva-default-gateway>
        set device port2
    next
end
```

### 5. Verify BGP sessions

From the FortiGate CLI:

```
get router info bgp summary
```

Both neighbors should show state **Established** with a non-zero uptime.

You can also verify from the **Azure Portal → Virtual Hub → BGP Peers** — both NVAs should show **Connected**.

> **This challenge is scored.** The prober checks Azure's reported BGP connection state for both NVAs. Both sessions must be **Connected** to score the challenge — if only one is connected you will receive a warning instead.
