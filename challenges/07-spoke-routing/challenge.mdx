The spoke VNet is now connected to the hub, but traffic between the spoke and the branch is currently bypassing the FortiGate NVAs — it routes through Azure's native router. This challenge configures **Routing Intent** to steer all private traffic through your FortiGates.

## Topology goal

```
Branch ──[IPsec]──► NVA 1/2 ──[policy check]──► Spoke server
                      ▲
               Routing Intent
               forces all traffic
               through NVAs
```

## Steps

### 1. Enable Routing Intent on the hub

Navigate to **Virtual Hub → Routing intent and routing policies**.

Enable **Private traffic** intent and set the next hop to your NVA resource:

| Setting | Value |
|---------|-------|
| Private traffic | Enabled |
| Next hop resource | FortiGate NVA (select your NVA) |
| Internet traffic | Leave disabled for now |

Click **Save**. Azure will take 2–5 minutes to propagate the new routing policy.

### 2. Create a firewall policy on the NVAs

With routing intent enabled, **all** traffic flows through the NVAs. Without a policy allowing it, traffic will be dropped. On each hub NVA, create a policy:

| Parameter | Value |
|-----------|-------|
| Name | `spoke-to-branch` |
| Incoming interface | hub internal interface |
| Outgoing interface | hub internal interface |
| Source | <EnvVarInline field="spoke_cidr" /> |
| Destination | <EnvVarInline field="branch_cidr" /> |
| Action | Accept |
| Log | All sessions |

Add a matching policy for the reverse direction (`branch-to-spoke`).

### 3. Test connectivity

From the **Windows Desktop** on the branch (<EnvVarInline field="branch_win_pip" />), open a command prompt and ping the spoke server:

```
ping <EnvVarInline field="spoke_server_private" />
```

The ping should succeed. If it does not, check the FortiGate traffic log for drops.

You can also verify routing from the branch FortiGate CLI:

```
get router info routing-table all
```

The spoke CIDR <EnvVarInline field="spoke_cidr" /> should appear as a BGP-learned route via the tunnel interfaces.
